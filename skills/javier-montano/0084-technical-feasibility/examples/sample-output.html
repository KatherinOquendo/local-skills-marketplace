D1: Technical Feasibility — 4/5

Core architecture patterns (DDD, event-driven, cloud-native) are proven in banking. PostgreSQL handles projected volume. Main risk is Kafka integration with legacy 2-phase commit patterns and 47 Oracle stored procedures requiring migration.

D2: Organizational Feasibility — 3/5

Team of 12 has strong Java/Spring skills but zero Kubernetes, limited event-driven, and no DDD experience. Conway’s Law alignment requires restructuring into 3 domain teams. Hire 1 senior platform engineer + contract DDD coach for Sprints 0-1.

D3: Timeline Feasibility — 3/5

Proposed 9-month timeline is aggressive given skill gaps. Critical path runs through K8s setup + first domain extraction + data migration. Recommend 11-month timeline with Sprint 0 buffer for spikes.

D4: Financial Feasibility — 4/5

Effort magnitudes align with industry benchmarks. Cloud cost projections validated by AWS TCO calculator. Hidden costs from hiring and extended timeline are manageable within contingency.

D5: Regulatory Feasibility — 4/5

Target architecture supports PCI-DSS and SOC 2. Data residency satisfied by AWS region selection. Audit trail maintained through event sourcing. Cloud certification is a parallel activity.

D6: Operational Feasibility — 3/5

Operations team has zero experience with K8s monitoring, distributed tracing, or event-driven debugging. Current stack is Nagios + custom scripts. Requires Grafana + Prometheus + Jaeger in Sprint 0.
