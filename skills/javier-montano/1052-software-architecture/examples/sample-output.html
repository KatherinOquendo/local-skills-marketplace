S3: Design Patterns

Hexagonal Architecture (Ports & Adapters)

Why chosen: Core Banking integrates with 4 external systems. Business logic must remain stable regardless of external changes.

  • Ports (inbound): PaymentService, AccountService, ComplianceService
  • Ports (outbound): LedgerPort, NotificationPort, AMLScreeningPort
  • Adapters (inbound): REST controllers, gRPC handlers, event consumers
  • Adapters (outbound): SwiftAdapter, CoreLedgerAdapter, EmailAdapter
CQRS — Transaction Processing

Why chosen: 50K+ events/day. Reporting queries degrade write performance when sharing the same model.

  • Command side: PaymentRepository appends domain events to event store
  • Query side: PaymentProjection builds read models from events (PostgreSQL materialized views)
  • Sync: Async projection via Kafka consumer with at-least-once delivery

S5: Architecture Decision Records

ADR-001: Hexagonal Architecture for Core Banking Domain Accepted

Context: 4 external systems with different protocols, SLAs, and change cycles. Business logic must remain stable.

Decision: Adopt Hexagonal Architecture with explicit port interfaces for all external integrations.

  • Domain logic independently testable — unit tests < 2s
  • External provider changes require only adapter modifications
  • More interfaces and classes than layered approach
  • Team requires training on ports & adapters

Alternatives: Layered Architecture (rejected: coupling to infra changes), Clean Architecture (rejected: heavier ceremony than needed).

ADR-002: CQRS for Transaction Processing Accepted

Context: 50K+ events/day. Reporting queries degrade write performance on shared model.

Decision: Separate command (event store) and query (materialized views) models for Payments module.

  • Write throughput increased 3x in load testing
  • Reporting queries p95 < 100ms
  • Eventual consistency: read model lags 200-500ms
  • Projection failure requires replay mechanism

Alternatives: Single model with read replicas (rejected: complex joins), Full Event Sourcing (deferred to Phase 3).

ADR-003: Kafka for Domain Event Distribution Accepted

Context: Multiple modules react to payment events. Synchronous calls create cascading failure risk.

Decision: Use Kafka as event bus for inter-module domain events with independent consumer groups.

  • Loose coupling — Compliance failure does not block Payments
  • Event replay enables rebuilding read models
  • Operational overhead: cluster management, partition strategy
  • At-least-once delivery requires idempotent consumers

Alternatives: RabbitMQ (rejected: no event replay), Direct async calls (rejected: no durability).

Evolution Strategy

Phase 1 — Sprints 1-3

Extract services from LegacyTransactionService. Introduce AMLScreeningPort for Compliance. Testing coverage target: 80% domain layer.

Phase 2 — Sprints 4-5

Implement Strategy pattern for payment types. Build event replay mechanism for projections. Consumer lag dashboards and projection health checks.

Phase 3 — Sprints 6-8

Evaluate Event Sourcing for full audit trail. Align module boundaries with team topology. Architecture fitness functions in CI pipeline.
