3. Architecture Decision Records

ADR-001: Saga Pattern (Orchestration) for Distributed Transactions

Accepted — 2026-03-12

Context: 50K+ daily transactions across Account, Payment, and legacy Core Banking require eventual consistency with compensation on failure.

Decision: Saga pattern with orchestration — Transaction Orchestrator service coordinates steps.

OptionProsConsVerdict
2PCStrong consistencyBlocking, poor at scaleRejected
ChoreographyLoose couplingHard to debug, implicit flowRejected
OrchestrationExplicit flow, centralized monitoringOrchestrator couplingAccepted

Consequences: (+) Clear transaction flow; (+) Testable compensation; (-) Orchestrator is critical path (HA required); (-) ~15ms latency per step.

ADR-002: Zero Trust Security Model (mTLS + OAuth2)

Accepted — 2026-03-12

Context: Banking regulations require encryption in transit, audit of all access, zero implicit trust.

Decision: Every service-to-service call uses mTLS; user-facing calls validated via OAuth2 at gateway AND service boundary.

OptionProsConsVerdict
Network perimeter onlySimpleLateral movement riskRejected
API Keys per serviceEasyNo rotation, no identityRejected
Zero TrustDefense in depthCertificate managementAccepted

Consequences: (+) Regulatory compliance; (+) No lateral movement; (-) Cert rotation ops (automate via cert-manager).

ADR-003: Event Sourcing for Audit Trail

Accepted — 2026-03-12

Context: Compliance requires immutable, queryable audit trail. Traditional logging insufficient — auditors need point-in-time reconstruction.

Decision: Event Sourcing for Audit domain only. Core services use CRUD + outbox pattern. Audit Service consumes all events into append-only Elasticsearch.

OptionProsConsVerdict
Full Event SourcingComplete historyMassive complexityRejected
CRUD + app logsSimpleMutable, audit gapsRejected
Hybrid (chosen)Audit-grade trailTwo consistency modelsAccepted

Consequences: (+) Immutable trail satisfies regulators; (+) Point-in-time replay; (-) ~2s eventual consistency lag; (-) ES cluster sizing required.

4. Nightmare Scenarios

Nightmare 1: Ghost Transaction

Problem: Payment commits in Payment Service but Saga compensation fails in Account Service. Customer charged but balance unchanged.

Mitigations (Defense in Depth)

  1. Idempotency keys on every saga step
  2. Outbox pattern — payment event in same DB transaction
  3. Dead Letter Queue — failed compensations + alert
  4. Reconciliation service — nightly Payment vs Account comparison
  5. Immutable audit log — forensic reconstruction
  6. Manual resolution runbook — admin API replay/compensate

Monitoring

  • Alert: Saga step pending > 60s
  • Alert: DLQ depth > 0
  • Alert: Reconciliation delta > $0.01

Acceptance: Zero ghost transactions in production. DLQ resolved within 4 hours.

Nightmare 2: Auth Service Outage

Problem: Auth Service unreachable — all API requests rejected at gateway, complete platform outage.

Mitigations (Defense in Depth)

  1. Token caching (Redis) — 5-minute grace period
  2. Multi-replica — 3 replicas across AZs
  3. Circuit breaker — gateway falls back to cached validation
  4. LDAP fallback — emergency auth source
  5. K8s liveness probe — auto-restart within 30s
  6. Emergency bypass — pre-shared ops tokens (audit-logged)

Monitoring

  • Alert: Error rate > 1% (5-min window)
  • Alert: P99 latency > 200ms
  • Alert: Certificate expiry < 7 days

Acceptance: Platform operational for 5 minutes during complete Auth failure via cached tokens.

Nightmare 3: Cascade Failure

Problem: Payment Service crashes. Retry storm floods Kafka. Account Service consumer falls behind. Full platform degradation.

Mitigations (Defense in Depth)

  1. Circuit breaker — open after 5 failures, half-open at 30s
  2. Bulkhead pattern — dedicated thread pools per dependency
  3. Rate limiting + backpressure — Kafka consumer pause
  4. Service mesh retry policy — max 3, exponential backoff
  5. K8s resource limits — prevent OOM cascade
  6. Load shedding — gateway rejects low-priority when error > 5%

Monitoring

  • Alert: Circuit breaker open on any service
  • Alert: Kafka consumer lag > 10K messages
  • Alert: Gateway 5xx rate > 2%

Acceptance: Single service failure does not degrade others. Recovery within 5 minutes.

CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE user_roles (
    user_id UUID REFERENCES users(id),
    role VARCHAR(50) NOT NULL,
    granted_at TIMESTAMPTZ DEFAULT now(),
    PRIMARY KEY (user_id, role)
);

CREATE TABLE session_events (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    event_type VARCHAR(30) NOT NULL,
    ip_address INET,
    user_agent TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);

6. Phased Migration — Strangler Fig

Phase 1: Assessment & Wrapping (Months 1–2)

  • Document legacy integrations, data flows, implicit business rules
  • Build ACL adapters for Core Banking (COBOL/AS400)
  • Deploy sidecar pattern for legacy LDAP
  • Establish observability baseline

Phase 2: MVP Deployment (Months 3–5)

  • Deploy Auth Service with 10% canary
  • Shadow mode for 2 weeks: compare both paths
  • Promote to 100% after zero-delta validation

Phase 3: Core Service Migration (Months 6–10)

  • Deploy Account, Payment, Transaction Orchestrator
  • Implement Saga + Kafka event streaming
  • Nightly reconciliation validates consistency

Phase 4: Legacy Sunset (Months 11–14)

  • Core Banking set to read-only
  • Archive to S3 Data Lake
  • Maintain ACL for compliance queries only
  • Decommission after 90-day observation
