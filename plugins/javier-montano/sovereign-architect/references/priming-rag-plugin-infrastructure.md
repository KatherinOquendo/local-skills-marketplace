## Hooks Architecture

SA uses Claude Code hooks defined in `hooks/hooks.json` with two event types:

### SessionStart Hook Chain

Executes sequentially when a new session begins:

```
secrets-scan.sh → scan-artifacts.sh → session-init.sh → auto-prime-check.sh → ghost-menu-inject.sh → session-changelog.sh
```

| Step | Script | Purpose | Blocks on Failure |
|------|--------|---------|-------------------|
| 1 | `secrets-scan.sh` | G0 Security Gate — scans for exposed secrets | Yes (hard stop) |
| 2 | `scan-artifacts.sh` | Detects previously generated artifacts | No |
| 3 | `session-init.sh` | Detects tech stack, generates initial context | No |
| 4 | `auto-prime-check.sh` | Checks if repo has priming-rag files | No |
| 5 | `ghost-menu-inject.sh` | Prepares contextual navigation menu | No |
| 6 | `session-changelog.sh` | Initializes action log | No |

### PostToolUse Hook Chain

Executes after Write or Edit operations:

```
session-changelog.sh → context-prune.sh → post-prime-calibrate.sh
```

| Step | Script | Purpose |
|------|--------|---------|
| 1 | `session-changelog.sh` | Logs the action with timestamp |
| 2 | `context-prune.sh` | Compresses context if >150 lines accumulated |
| 3 | `post-prime-calibrate.sh` | Recalibrates if priming-rag files were modified |

## sa/ Session Directory

Generated automatically in the working directory. Provides session state and continuity.

| File | Purpose | Generated By |
|------|---------|-------------|
| `SESSION-README.md` | Repository context (stack, structure, signals detected) | `session-context-gen.sh` |
| `SESSION-CLAUDE.md` | Session-specific instructions (curated from CLAUDE.md) | `session-context-gen.sh` |
| `calibration-digest.md` | Priming state (what SA knows about the repo) | `post-prime-calibrate.sh` |
| `ghost-menu.md` | Contextual navigation menu | `ghost-menu-inject.sh` |
| `session-changelog.md` | Timestamped log of all actions | `session-changelog.sh` |

## Quality Gate Mechanics

### G0 — Security Gate (Session Start)

- **Trigger**: Automatic on SessionStart
- **Script**: `secrets-scan.sh`
- **Scans for**: `.env` files with credentials, hardcoded API keys, private keys, tokens in source
- **On failure**: Session blocked until secrets are remediated
- **Output**: `.sa-secrets-audit.log`

### G1 — Discovery Gate (Post-Diagnosis)

- **Trigger**: Transition from Phase 2 to Phase 3
- **Criteria**: 100% evidence tagged, risks scored, gaps declared, <30% assumptions
- **On failure**: No design proposals — request more information or declare limitations

### G1.5 — Design Gate (Post-Design)

- **Trigger**: Transition from Phase 3 to Phase 4
- **Criteria**: Minimum 2 alternatives, justified decisions, scope verified, constraints mapped, diagram present
- **On failure**: Revise design or request more constraints

### G2 — Delivery Gate (Post-Delivery)

- **Trigger**: Before final delivery to user
- **Criteria**: All artifacts complete, tests proposed, rollback defined, docs updated, excellence loop passed
- **On failure**: Complete missing artifacts before delivery

## Progressive Loading System

The `lazy-load-resolver.sh` determines loading depth:

| Level | Content Loaded | ~Lines/Skill | When |
|-------|---------------|-------------|------|
| **L1** | Frontmatter + TL;DR only | ~20 | Always (minimum viable context) |
| **L2** | Full SKILL.md | ~200 | Skill is activated by signal |
| **L3** | SKILL.md + references/ + examples/ | ~500+ | Deep dive explicitly requested |

Resolution logic:
1. Check current phase and movement
2. Match detected signals to skill activation rules
3. Load at L1 for awareness, L2 for active use, L3 for deep analysis
