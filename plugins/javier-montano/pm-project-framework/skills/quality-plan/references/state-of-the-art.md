# State of the Art — Quality Plan (2024-2026)

## Trend 1: Quality-as-Code

Quality plans encoded as machine-readable specifications integrated into CI/CD pipelines. Quality gates, metrics thresholds, and compliance checks are automated and enforced at build time. [DOC]

> "Principio Rector: La calidad planificada es calidad garantizada."

## Trend 2: Shift-Left Quality Planning

Quality planning moved to the earliest project phases, with quality criteria embedded in user story acceptance criteria, architecture decision records, and design specifications. [INFERENCIA]

| Phase | Traditional Quality | Shift-Left Quality |
|-------|-------------------|-------------------|
| Requirements | Quality reviewed after | Quality criteria co-defined |
| Design | Quality checkpoint | Quality built into architecture |
| Development | Test after code | TDD/BDD quality first |
| Deployment | Post-deploy testing | Quality gates in pipeline |

## Trend 3: Adaptive Quality Plans

Static quality plans replaced by adaptive plans that adjust quality activities based on risk profiles and historical defect patterns. High-risk components receive more rigorous quality attention. [DOC]

## Trend 4: AI-Enhanced Quality Prediction

ML models predict quality outcomes based on code complexity, change frequency, developer experience, and historical defect data. Quality plans dynamically allocate testing effort based on predictions. [INFERENCIA]

## Trend 5: Customer-Centric Quality Metrics

Traditional internal quality metrics (defect count, coverage) supplemented with customer experience metrics (NPS, task success rate, time-on-task) as primary quality indicators. [DOC]

| Metric Type | Traditional | Customer-Centric |
|------------|-------------|-----------------|
| Code quality | Defect density | Customer-reported issues |
| Test quality | Coverage % | User acceptance rate |
| Process quality | Audit compliance | Time to value |
| Design quality | Architecture review score | Usability score |
