Deuda Tecnica
y Riesgos

V&A comienza con cero deuda tecnica tradicional (no hay legacy code, no hay tech stack desfasado). Pero enfrenta riesgos especificos a (R)Evolución AI-Native: prompt debt y vendor lock-in.

Prompt Debt

A medida que agentes evolucionan, prompts acumulan logica, edge cases, workarounds. Sin governance estricta, prompts se vuelven oscuros, dificiles de debuggear, costosos de mantener.

Mitigacion: Versioning enforced. Template prompts reutilizables. Rubrics explicitos. Code review de prompts (peer reviewer). Documentation clara de intencionalidad.

Vendor Lock-In (Google Workspace)

100% Google minimiza complexity pero creaa dependencia. Si V&A necesita migrar data o agentes a otra plataforma, requiere translation de prompts, re-indexing de sources, reconfiguracion de Opal flows.

Mitigacion: Prompts almacenados en texto plain (exportables). Knowledge bases en formato standard (Markdown, CSV). Opal workflows documentados en diagramas. Export paths planificados desde inicio.

Escalabilidad de Agentes

Cada agente adicional aumenta complejidad del pipeline, numero de integraciones, superficie de testing. El sistema necesita coordinacion entre 6 agentes sin crear cascadas de fallo (si Asistente Propuestas falla, Task Tracker no deberia bloquearse).

Mitigacion: Event decoupling. Fallback paths. Independent versioning por agente. Monitoring de agent health.

Prompt Debt
as Technical Debt

Risk Management
What is Prompt Debt?

Over time, prompts accumulate: edge cases ("if client is named Acme, use X template"), workarounds ("when Gemini hallucinates, try rephrasing"), hacks ("prepend fake data to ground context"). Prompt becomes unmaintainable.

How to Avoid It

1. Template prompts reutilizables (copy from library, no mutations). 2. Explicit rubrics (define success criteria upfront). 3. Versioning enforced (v1.1 is iteration of v1.0, not patch). 4. Code review de prompts (Peer Reviewer validates). 5. Testing before release.

Mitigation Strategy

Monthly prompt audit: revisar prompts viejos, identificar cruft, consolidar workarounds en nueva version. Limit agent to 1-2 major prompts max. If > 2, split into sub-agents.

Success delivered

Prompts permanecen legibles, debuggeables, efficientes. Agentes mantienen calidad a largo plazo sin technical debt spiral.
