Pipeline de Discovery

16 Entregables, 4 Quality Gates, Protocolo Zero-Hallucination

¿Qué es un Quality Gate?

Un gate es un "alto obligatorio" en el pipeline. Si no pasas el criterio de aceptación, el proceso se detiene hasta validar con el cliente. No se continúa con entregables posteriores sin resolver.

Flujo Visual: 16 Fases + 4 Gates

00 Plan ↓ 01 Stakeholders → 02 Brief → 03 AS-IS → 04 Flows ↓ ┌─── GATE 1 ───┐ ↓ ↓ 05 Escenarios → 05b Feasibility ↓ (7 Sabios) ┌─ GATE 1.5 ─┐ ↓ ↓ 06 Roadmap → GATE 2 ↓ 07 Spec → 08 Pitch → 09 Handover ↓ GATE 3 (Final)

Decisor: ¿Cuál Pipeline Usar?

Modo Duración Entregables Cuándo Usar
express 1-2 horas 3 (Brief, AS-IS, Escenarios) Go/No-Go rápido, cliente apremiado, validación preliminar
guided 1-2 días 9 (00-09) Conversación humana, sesiones facilitadas, iteración cliente
auto 4-8 horas 9 (00-09) Discovery autónomo, sin intervención, cliente review final
deep 3-5 días 14 (00-14 + reportes) Dirección arquitectónica, decisiones críticas, análisis exhaustivo

Quality Gates: Criterios Duros

Gate Fase Criterios de Pase ¿Hard Stop? Evidencia Requerida
G1 Post-Escenarios Scoring 6D completo, escenario recomendado justificado, 3+ opciones evaluadas SÍ Matriz decisional, trade-offs documentados
G1.5 Post-Feasibility 7 Sabios: veredicto Go (≥5/7) o No-Go, con justificación por dimensión SÍ Acta de 7 Sabios, 2+ riesgos mitigados
G2 Post-Roadmap FTE-meses validados, Monte Carlo ejecutado, 3+ quick wins definidos SÍ Roadmap de 18-24 meses, sprints fase 1 desagregados
G3 Final (Handover) Consistencia cruzada, excellence-loop pasado (10 criterios), changelog completo SÍ Cross-references entre 09-14, sign-off stakeholders

Casos Especiales: Sin Codebase

¿Qué pasa si el cliente NO tiene codebase?

Fase 03 (AS-IS) se adapta: procesos actuales, stack de herramientas, integración manual. Fase 04 (Flows) documenta flujos de negocio en lugar de flujos de código. Las skills de Data & Architecture se orientan a greenfield. Roadmap propone tecnología desde cero en lugar de migraciones.

Casos Especiales: Solo Documentación

¿Qué pasa si solo hay documentación, sin acceso al código?

Fase 03 se basa en docs arquítectonicos. Assessment es [SUPUESTO] high-risk: requiere validación posterior con acceso. Se marcan secciones con tag [INFERENCIA] o [SUPUESTO] según el protocolo ZH.

¿SDA vs Digital Transformation?

SDA: Enfoque técnico puro. Arquitectura, integración, tecnología. Supone que el negocio ya sabe qué quiere.

Digital Transformation: Descubrimiento integral. Procesos → tecnología → cambio. Valida hipótesis de negocio, rediseña procesos. Mucho más profundo y largo.

Multi-Service: Combos Comunes

Data-AI + SDA
Arquitectura + pipelines de datos + modelos. Típico en banking/retail.
Cloud + SDA
Migración a cloud + diseño de infraestructura. Common enterprise move.
Digital-Transformation + QA
Cambio de procesos + testing strategy. Compliance-heavy.
Management + UX-Design
Transformación + rediseño de experiencia. Customer-centric.

Modelo de Voto

Consenso: ≥5/7 Go = PROCEED

Si 5+ Sabios votan "Go", el proyecto continúa. Si <5, se marca NO-GO y se documentan riesgos críticos. Cada Sabio puede emitir condiciones: "Go si..." (e.g., "Go si presupuesto +20%").

¿Qué es MOAT?

MOAT = Modernization-Oriented Acceleration Technology. En MetodologIA —cuyo nombre nace de SOFTWARE + KAIZEN— la modernización y la mejora continua son ADN. Los skills MOAT son propiedad intelectual que NO están disponibles en ChatGPT o Claude base: entrenamientos especializados, templates verificados y workflows operativos construidos desde la experiencia real de +1.200 Sofkianos en 8 países. Cada skill codifica conocimiento acumulado en una barrera competitiva (moat) que hace de MAO un diferenciador único.

Protocolo Zero-Hallucination

Etiquetado de Evidencia en Cada Afirmación

Regla Absoluta

Toda afirmación en un entregable DEBE estar etiquetada con su fuente de evidencia. Si no tiene evidencia clara → se marca [SUPUESTO] + requiere validación posterior.

Tabla de Tags de Evidencia

Tag Significado Confianza Ejemplo
[CÓDIGO] Verificado en código fuente Alta "La API usa GraphQL [CÓDIGO: main/api/gateway.ts]"
[CONFIG] Verificado en configuración Alta "Timeout 30s [CONFIG: docker-compose.yml]"
[DOC] Documentado en fuentes del proyecto Alta "Soporta OAuth2 [DOC: docs/security.md]"
[INFERENCIA] Razonado desde patrones observados Media "Probablemente usa JWT [INFERENCIA: estructura de tokens en logs]"
[SUPUESTO] Asunción no verificable Baja "Se asume 2k TPS capacity [SUPUESTO: requiere load-test]"
[STAKEHOLDER] Declarado por stakeholder Variable "Migración en Q3 [STAKEHOLDER: CEO roadmap]"

Reglas de Aplicación

  • Si NO puedes etiquetar una afirmación → NO la incluyas sin marcar [SUPUESTO].
  • Si >30% de un entregable es [SUPUESTO] → emitir banner de advertencia al inicio.
  • Si un hallazgo es CRÍTICO + [SUPUESTO] → flag obligatorio para validación con stakeholder.
  • NUNCA presentar [INFERENCIA] como hechos verificados.
  • Ante duda → siempre [SUPUESTO] con nota: "requiere validación con {stakeholder}".

Ejemplo: Entregable 03 (AS-IS) con Etiquetado

ARCHITECTURE [CÓDIGO: repo-index, archi.md] ├── Frontend: React 18.2 [CÓDIGO: package.json] ├── Backend: Node.js 18 [CONFIG: Dockerfile] ├── DB: PostgreSQL 14 [CONFIG: docker-compose.yml] ├── Cache: Redis [SUPUESTO: requiere confirmar con ops] ├── API Gateway: Kong [DOC: infrastructure/diagrams.md] ├── Message Queue: RabbitMQ [INFERENCIA: declaraciones de broker en logs] └── Monitoring: Prometheus [STAKEHOLDER: DevOps lead confirms]

Checklist Pre-Entrega

  • ¿Cada afirmación técnica tiene tag [CÓDIGO], [CONFIG], [DOC], [INFERENCIA], [SUPUESTO]?
  • ¿Hay notas al pie para cada [SUPUESTO] indicando "requiere validación"?
  • ¿Porcentaje [SUPUESTO] < 30%?
  • ¿Hallazgos CRÍTICOS tienen evidencia [CÓDIGO] o [CONFIG], no [SUPUESTO]?
  • ¿changelog registra qué fue validado y qué requiere post-discovery?

Impacto en Credibilidad

Protocolo ZH diferencia a MAO Framework:

Los clientes ven EXACTAMENTE dónde vino cada afirmación. No es "confianza en Claude" — es "evidencia trazable en el código cliente". Aumenta credibilidad 10×. Permite que cliente VERIFIQUE de forma independiente.

Persistencia: Recuperación ante Crisis

Changelog + Session State = crash recovery.

Primeros Pasos

Setup, Prerequisitos, Troubleshooting

Pre-Discovery: Checklist de Entrada

Antes de `/mao:run-*`, asegúrate de:

  • Acceso a código fuente cliente (repo, URL, descarga)
  • Documentación disponible (README, architecture docs, roadmap)
  • Lista de stakeholders: roles, emails, horarios (si guided/supervisado)
  • Presupuesto / FTE disponible (para roadmap)
  • Objetivo claro: qué decisión necesita? (arquitectura, roadmap, go/no-go)
  • Restricciones: tecnología legacy, compliance, time-to-market

Paso a Paso: Pipeline Express (1-2h)

  1. Inicia sesión en Claude Code
    Carga el framework: `/mao:run-express --tipo-servicio=SDA`
  2. Proporciona el brief
    Comparte: descripción proyecto, 2-3 archivo claves (código o docs)
  3. Espera 02-Brief (15 min)
    Framework extrae contexto, genera brief técnico automático
  4. Revisión AS-IS (30 min)
    Fase 03: arquitectura actual, stack, dependencias, deuda técnica
  5. Escenarios & Go/No-Go (30 min)
    Fase 05: 3+ opciones evaluadas, scoring 6D, recomendación
  6. Entrega: Pitch + Documento (15 min)
    Resumen ejecutivo + profundización técnica. Listo para cliente.

Troubleshooting: Problemas Comunes

❌ "El discovery quedó atorado en fase X"

Ejecuta: /mao:rescue-stalled --fase=X. Framework diagnostica por qué el gate no pasó, sugiere mitigaciones.

❌ "Cliente no aprobó gate 1.5"

7 Sabios votaron No-Go en alguna dimensión. Revisa acta, valida la objeción (financiera, técnica, etc.), itera escenario, re-ejecuta.

❌ "Tengo >30% [SUPUESTO] en entregable"

Banner de advertencia aparece. Marca qué requiere validación, envía a cliente, pausa discovery hasta respuesta.

❌ "Sesión se cayó a mitad del discovery"

No hay problema. El changelog está en `.discovery/session-changelog.md`. Reinicia sesión, referencia el changelog, continúa desde donde se pausó.

❌ "Necesito cambiar {TIPO_SERVICIO} a medio camino"

Es posible. Ejecuta: /mao:improve-deliverables --nuevo-tipo=Data-AI. Framework actualiza agentes, re-evalúa deliverables, propone ajustes.

❌ "No tengo acceso al código"

Usa documentación disponible. Fase 03 marca todo como [SUPUESTO] o [INFERENCIA]. Pausa discovery, solicita acceso o código samples. No continúes con gates sin evidencia.

Session Persistence: Crash Recovery

Los 3 Archivos Mágicos de Recuperación

1. `.discovery/session-changelog.md` — Log de cada acción, timestamp, resultado. Si sesión cae, Lee esto primero.

2. `.discovery/session-state.json` — Estado actual: fase, gates pasados, entregables generados.

3. `.discovery/ghost-menu.md` — Dónde estamos ahora, qué sigue.

Manda estos 3 archivos en el siguiente prompt. Claude reconoce el contexto y continúa SIN repetir trabajo.

Consejos Prácticos

  • Usa {MODO}=piloto-auto para eficiencia máxima, pero {MODO}=supervisado si cliente quiere controlar cada entregable.
  • Si cliente es no-técnico, usa {VARIANTE}=ejecutiva (40% tamaño, resúmenes ejecutivos).
  • Guarda los 9 primeros entregables (00-09) antes de avanzar a reportes (10-14). Los reportes usan los anteriores como base.
  • Para proyectos complejos (multi-servicio), alloca 3-5 días con /mao:run-deep. Vale cada hora.
  • Siempre marca [SUPUESTO] si duda. Credibilidad >> velocidad.
