# Body of Knowledge — Input Analysis

## 1. Foundations of Input Analysis

Input analysis is the systematic evaluation of all inputs (documents, data, stakeholder communications, constraints) that inform project decisions. It ensures that planning and execution are based on validated, complete, and consistent information. [DOC]

### 1.1 Input Categories

| Category | Examples | Validation Method |
|----------|---------|-------------------|
| Strategic Inputs | Business case, strategic plan, OKRs | Alignment check |
| Stakeholder Inputs | Requirements, expectations, constraints | Interviews, workshops |
| Technical Inputs | Architecture docs, API specs, data models | Technical review |
| Financial Inputs | Budget, cost estimates, funding approval | Financial validation |
| Historical Inputs | Lessons learned, benchmarks, baselines | Relevance assessment |
| Environmental Inputs | Market data, regulations, competitor intel | Currency check |

### 1.2 Input Quality Dimensions

1. **Completeness** — Are all required inputs present? [METRIC]
2. **Accuracy** — Are inputs factually correct? [METRIC]
3. **Consistency** — Do inputs align with each other? [INFERENCIA]
4. **Currency** — Are inputs up to date? [METRIC]
5. **Relevance** — Do inputs apply to this project context? [INFERENCIA]
6. **Traceability** — Can inputs be traced to authoritative sources? [DOC]

## 2. Standards and Frameworks

### 2.1 PMBOK Input-Tool-Output Model

Every PMBOK process defines inputs, tools & techniques, and outputs. Input analysis validates that process inputs meet quality thresholds before proceeding. [DOC]

### 2.2 Requirements Engineering (IREB)

The International Requirements Engineering Board provides standards for requirements elicitation, analysis, and validation — core input analysis activities. [DOC]

### 2.3 Data Quality Standards (ISO 8000)

ISO 8000 provides a framework for data quality that applies to project input validation: completeness, accuracy, timeliness, and consistency. [DOC]

## 3. Input Analysis Techniques

| Technique | Application | Output |
|-----------|-------------|--------|
| Gap Analysis | Identify missing inputs | Gap register |
| Consistency Matrix | Cross-check input alignment | Conflict register |
| Source Credibility Assessment | Evaluate input reliability | Confidence scores |
| Assumption Identification | Extract implicit assumptions | Assumption log |
| Sensitivity Analysis | Determine which inputs matter most | Priority ranking |
| Historical Comparison | Compare against past project inputs | Benchmark report |

## 4. Input Triage Process

1. **Inventory** — Catalog all available inputs [PLAN]
2. **Classify** — Categorize by type and criticality [PLAN]
3. **Validate** — Check quality dimensions [METRIC]
4. **Reconcile** — Resolve conflicts and gaps [INFERENCIA]
5. **Prioritize** — Rank by impact on project decisions [PLAN]
6. **Document** — Record analysis results and confidence levels [DOC]

## 5. Stakeholder Input Management

| Stakeholder Type | Input Method | Validation Approach |
|-----------------|-------------|-------------------|
| Executive Sponsor | Strategic briefs, vision statements | Alignment workshop |
| End Users | Requirements, feedback, usage data | Prototype testing |
| Technical Teams | Architecture constraints, estimates | Peer review |
| Compliance | Regulatory requirements | Legal review |
| Vendors | Proposals, SLAs, technical specs | Due diligence |

## 6. Common Pitfalls

1. **Input Overload** — Too many inputs without prioritization [INFERENCIA]
2. **Stale Inputs** — Using outdated information for current decisions
3. **Single-Source Dependency** — Critical inputs from one unvalidated source
4. **Implicit Assumptions** — Unstated assumptions treated as facts
5. **Format Fragmentation** — Inputs in incompatible formats preventing analysis
